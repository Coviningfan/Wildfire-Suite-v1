1. Passwords Stored in Plain Text in AsyncStorage
File: stores/auth-store.ts

ts
const newUser = {
  id: ..., name, email,
  password,        // ← raw plain text string
  role: 'user',
};
await saveUsersToStorage([...users, newUser]);
All user passwords are serialized as plain text into AsyncStorage under the key 'local-users'. On a jailbroken/rooted device or via an unencrypted iTunes backup, any tool (iMazing, libimobiledevice) can read this key directly. The hardcoded demo and admin accounts also sit in the module's localUsers array. At minimum, passwords should be hashed with expo-crypto SHA-256 before storage.

3. initializeAuth() Is Synchronous But Zustand Persist Hydrates Asynchronously — True Root Cause of Login Flash
File: stores/auth-store.ts, app/_layout.tsx

ts
// auth-store.ts
initializeAuth: () => {
  const state = get();       // ← called before AsyncStorage hydration completes
  if (state.user) { ... }    // state.user is STILL null here
},

// _layout.tsx
useEffect(() => {
  initializeAuth();          // reads pre-hydration state (user: null)
  SplashScreen.hideAsync();  // fires immediately
}, [initializeAuth]);
AsyncStorage is asynchronous. Zustand's persist middleware hydrates state from AsyncStorage in a future microtask. When initializeAuth() runs synchronously, state.user is always the default null — hydration hasn't completed. The simple fix of await-ing initializeAuth() won't work because initializeAuth is not async. The real fix requires waiting for the persist rehydration event:

ts
// Correct implementation
useEffect(() => {
  const unsub = useAuthStore.persist.onFinishHydration(() => {
    initializeAuth();
    SplashScreen.hideAsync();
    unsub();
  });
  // also check if already hydrated synchronously (rare but possible)
  if (useAuthStore.persist.hasHydrated()) {
    initializeAuth();
    SplashScreen.hideAsync();
  }
}, []);
4. ResultCard.formatLabel Regex Broken for _mWm2 Keys — Core Results Screen Shows Irradiance MWm2
File: components/ResultCard.tsx

ts
const formatLabel = (key: string): string =>
  key
    .replace(/_/g, ' ')
    .replace(/\b\w/g, l => l.toUpperCase())  // → "Irradiance MWm2"
    .replace(/Mwm2/g, 'mW/m²')              // ← NEVER matches "MWm2"
After the title-case step, irradiance_mWm2 becomes Irradiance MWm2 (the W is mid-word so it stays uppercase W from the original camelCase). Fix: replace(/[Mm][Ww]m2/gi, 'mW/m²') or process before title-casing.

5. RoomSimulation Heatmap Uses Omnidirectional Point Source — Safety-Relevant Inaccuracy
File: components/RoomSimulation.tsx

ts
// sampleSurfaceCells and peopleReadouts both use:
totalIrr += (fixtureData.peak_irradiance_mWm2 * calibrationFactor) / (distance3D * distance3D);
The heatmap and person dose calculations treat each fixture as an omnidirectional point source emitting equally in all directions. In reality, UV fixtures have defined beam angles (often 40-120°). This model illuminates grid cells behind and above the fixture at full inverse-square intensity — physically impossible for a directional source.

The consequence: person markers positioned outside the actual beam cone receive irradiance values and DANGER/WARNING labels even though they're outside the beam. A venue operator could see a false DANGER reading for a person who is actually not in the beam. Conversely, the stats.safety in the bar (which uses beamData.irradiance from the proper model) may say SAFE while the person readout (using this formula) says DANGER. This creates contradictory safety signals.

Fix: Apply a beam cone check using beam_h_deg and beam_v_deg from fixture data before accumulating irradiance.

8. Tutorial FLAME Formula Uses mW/cm² — App Uses mW/m² — 10,000× Unit Discrepancy
File: constants/tutorials.ts

ts
// tutorials.ts FLAME formula section:
"I is incident UV irradiance (mW/cm²)"
"...standard blacklight tube at 1m = 0.5-1 mW/cm²"
"...professional Wildfire fixture at 1m = 5-15 mW/cm²"
The app calculator, safety thresholds, and all result displays use mW/m². The tutorials use mW/cm² for the FLAME formula. The conversion is 1 mW/cm² = 10,000 mW/m².

This means:

Tutorial's "safe" 0.5 mW/cm² = 5,000 mW/m² → CAUTION in the app

Tutorial's "professional fixture" at 5-15 mW/cm² = 50,000-150,000 mW/m² → deep DANGER in the app

A user reading the tutorials believes a typical Wildfire fixture at 1m produces "5-15 mW/cm²" which sounds reasonable — they cannot tell this is actually 2-6× above the DANGER threshold in the app's own safety system. This creates a false sense of safety in a UV exposure context.

Fix: Either convert all tutorial values to mW/m², or add an explicit unit conversion note at the start of the FLAME tutorial.
