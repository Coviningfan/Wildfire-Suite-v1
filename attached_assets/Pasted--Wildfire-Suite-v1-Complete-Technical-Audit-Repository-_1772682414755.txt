# Wildfire Suite v1 — Complete Technical Audit
**Repository:** `Coviningfan/Wildfire-Suite-v1`
**Audit Date:** March 4, 2026
**Stack:** Expo SDK 54 · React Native 0.81.5 · Expo Router v6 · Zustand · TypeScript
**Commit:** `441b528`

---

## Architecture Overview

The app is a React Native / Expo SDK 54 app using Expo Router v6 for file-based navigation, with a `(tabs)` group containing 6–7 tabs. State is managed entirely by Zustand with `persist` middleware writing to `AsyncStorage`. There is no backend — all auth, calculations, and data are local-only. The core calculation engine (`LightingCalculator`) is a pure TypeScript class with no external dependencies.

---

## 🔴 P0 — Ship-Blocking / Critical Security

### 1. Missing `expo-camera` Plugin → iOS App Store Rejection
**File:** `app.json`

`expo-camera` is used for QR scanning but is entirely absent from the `plugins` array. In managed Expo workflow, `NSCameraUsageDescription` is never written to `Info.plist`. Apple will reject the build at App Store review. On-device, the camera permission dialog shows a blank or generic description string, which iOS may silently deny.

**Fix:**
```json
["expo-camera", { "cameraPermission": "Wildfire uses the camera to scan fixture QR codes." }]
```

---

### 2. Missing `expo-local-authentication` Plugin → Face ID Hard Crash
**File:** `app.json`

`expo-local-authentication` requires `NSFaceIDUsageDescription` in `Info.plist`. Without the plugin declared, calling `authenticateAsync()` on any Face ID device throws a native exception — not a JS error — so no error boundary can catch it. The app crashes entirely.

**Fix:**
```json
["expo-local-authentication", { "faceIDPermission": "Wildfire uses Face ID to securely sign you in." }]
```

---

### 3. Rork Branding in All Production Identifiers
**File:** `app.json`

All four of the following are Rork defaults that have never been changed:
- `"scheme": "rork-app"` — every deep link uses `rork-app://`
- `"bundleIdentifier": "app.rork.wildfire-app-demo"` (iOS)
- `"package": "app.rork.wildfire_app_demo"` (Android)
- `expo-router origin: "https://rork.com/"`

Every App Store listing, Play Store listing, and device deep link will carry Rork's namespace. This cannot be changed post-submission without losing existing installs.

**Fix:** Replace all four with Wildfire-branded values before any production build.

---

### 4. Passwords Stored in Plaintext in AsyncStorage
**File:** `stores/auth-store.ts`

All user passwords are serialized to JSON and stored unencrypted in AsyncStorage under `'local-users'`. On Android, AsyncStorage is unencrypted by default — on a rooted device these are fully readable. The data structure:
```ts
{ id, name, email, password: "password123", role, createdAt }
```
is stored verbatim to disk.

**Fix:** Hash passwords with `expo-crypto` (`Crypto.digestStringAsync(Crypto.CryptoDigestAlgorithm.SHA256, password)`) before storing. Never store the raw string.

---

### 5. Hardcoded Credentials Committed to Public Repo
**File:** `stores/auth-store.ts`

`demo@example.com / password123` and `admin@example.com / admin123` are hardcoded directly in the store source and visible to anyone who views or forks the repository. These credentials also seed the admin role check.

**Fix:** Move to `.env` variables or remove entirely and use a `DEMO_MODE` flag separate from the auth flow.

---

### 6. Apple Sign In Stores Blank Password — Auth Bypass
**File:** `stores/auth-store.ts`

When a new Apple user registers, `password: ''` is written to the users array:
```ts
existingUser = { id, name, email, password: '', role: 'user', ... };
```
This means `login("apple@privaterelay.com", "")` successfully authenticates that account via the standard email/password flow — a complete bypass for any Apple-registered account.

**Fix:** Flag Apple accounts with `authProvider: 'apple'` and exclude them from password-based login entirely.

---

### 7. File Writes Not Awaited → Every Export Shares an Empty File
**File:** `utils/file-helpers.ts`

The `expo-file-system` new File API requires `await` on `.create()` and `.write()`. Neither is awaited across all three export functions:
```ts
file.create({ overwrite: true });   // missing await
file.write(content);                // missing await
await Sharing.shareAsync(file.uri); // shares empty/partial file
```
This affects `exportCalculationAsText`, `exportCalculationAsCSV`, and `exportTechSheet`.

**Fix:**
```ts
await file.create({ overwrite: true });
await file.write(content);
```

---

### 8. Imperial Inputs Exported Labeled as Meters
**File:** `utils/file-helpers.ts`

CSV headers hardcode `(m)` for all distance/area columns. The text report hardcodes `m` as the unit label on every input field. `buildCalculationReport` accepts no unit system parameter — there is no mechanism to pass it even if you wanted to fix it. If the user works in imperial, exported data shows feet values labeled as meters. This is a professional data integrity failure with direct safety implications.

**Fix:** Add `unitSystem: 'metric' | 'imperial'` to `buildCalculationReport`'s signature; apply `convertDistance`/`convertArea` from `settings-store.ts` to all output values.

## 🟠 P1 — High Severity (Crashes / Wrong Data)

### 9. React 19 + React Native 0.81 Version Mismatch
**File:** `package.json`

React Native 0.81 officially supports React 18.x. React 19.1.0 introduces concurrent rendering behaviors, `use()` hooks, and ref-as-prop changes unsupported in the RN 0.81 renderer. This causes silent rendering failures and undefined behavior in concurrent mode edge cases.

**Fix:** Downgrade to `"react": "18.3.1"` and `"@types/react": "~18.3.0"` until RN 0.82+ is adopted.

---

### 10. `newArchEnabled: true` with Incompatible Packages
**File:** `app.json`

New Architecture (Fabric/JSI/TurboModules) is enabled. Three packages have known issues at their pinned versions:
- `react-native-worklets: 0.5.1` — alpha, predates stable New Arch support
- `expo-sensors` LightSensor — known issues on New Arch with older Android driver stacks
- `react-native-gesture-handler ~2.28` — Fabric issues on specific Android OEMs

Result: device-specific crashes that are impossible to reproduce in CI.

**Fix:** Disable `newArchEnabled` until all packages confirm New Arch compatibility, or pin to explicitly certified versions.

---

### 11. `xPos`/`zPos` String-in-Numeric Type Corruption → SVG NaN
**File:** `stores/simulation-store.ts`

`updateFixture(id, field, value)` accepts `field: keyof SimFixture` and `value: string`. But `SimFixture.xPos` and `zPos` are typed `number | undefined`. Any call:
```ts
updateFixture(id, 'xPos', '5.0')
```
stores the string `"5.0"` in a field typed as `number`. The 47KB `RoomSimulation.tsx` uses these in SVG coordinate arithmetic — string arithmetic (`"5.0" - 2`) produces `NaN`, silently breaking SVG rendering. This is the root cause of the division-by-zero bug patched in commit `59d2b05`.

`addBlankFixture` also leaves `xPos` and `zPos` as `undefined`, causing new fixtures to render at `(0,0)` or disappear entirely until manually positioned.

**Fix:** Remove `xPos`/`zPos` from the `updateFixture` keyof union. Use the existing `updateFixturePosition(id, xPos: number, zPos: number)` exclusively.

---

### 12. `ErrorBoundary` Component Exists But Is Never Used
**File:** `components/ErrorBoundary.tsx`

A full `ErrorBoundary` class component exists but is never imported or used anywhere. If `LightingCalculator.calculateRadiometricData` throws, if the AI response parse fails, or if `RoomSimulation.tsx` hits a NaN in SVG math, the entire app crashes to a white screen with no recovery path.

**Fix:** Wrap the calculator screen, AI tab, and simulator each in `<ErrorBoundary>`. The component is already written — it just needs to be applied.

---

### 13. EM Series Tech Sheet States Incorrect Control Type
**File:** `utils/file-helpers.ts`

`getFixtureTechSheetContent` outputs `Control: On/Off (Mains)` for all EM-series fixtures. The EM-44V and EM-43E are DMX fixtures per the uploaded TDS documents (`TDS_Effects_Master_LED_4-_Four_Lamp_215-079-05.pdf`). This incorrect data is exported and shared with clients as an official Wildfire tech sheet.

**Fix:** Split the EM branch: `EM-44V` / `EM-43E` → DMX 512/RDM; `EM-44L` / `EM-42L` / `EM-22L` / `EM-42E` → mains.

---

### 14. `react-native-worklets: 0.5.1` Is an Alpha Dependency
**File:** `package.json`

This is an early alpha of `react-native-worklets-core`. It may conflict with the existing `Animated` API usage throughout the codebase and with New Arch. Unless something explicitly requires it, this is likely a transitive dependency mistake that was never audited.

**Fix:** Audit whether anything actually imports from this package. If not, remove it.

## 🟡 P2 — Medium (Wrong Behavior / Broken UX)

### 15. Three Independent Copies of Safety Threshold Data
**Files:** `types/lighting.ts`, `stores/lighting-store.ts`, `utils/file-helpers.ts`

`SAFETY_THRESHOLDS` and `SAFETY_LABELS` are exported from `types/lighting.ts` with full threshold values and guidance strings ("Limit exposure to 5 min", "Immediate evacuation required"). Neither is ever imported anywhere. The store uses its own inline numbers. The home screen has its own hardcoded `safetyColors` map. `file-helpers.ts` has a third raw-string copy.

If thresholds change, two of three copies will silently serve wrong data.

**Fix:** Delete all inline copies. Import exclusively from `types/lighting.ts`.

---

### 16. OS Theme Preference Ignored on First Launch
**File:** `stores/settings-store.ts`

`themeMode` defaults to `'dark'` regardless of system setting. `app.json` sets `"userInterfaceStyle": "automatic"`, but the app reads `themeMode` from Zustand, not `useColorScheme()`. A user whose phone is in light mode gets dark mode on first open. There is no first-launch initialization from the OS.

The `convertToMetric` function is also exported from this file but never called anywhere — dead code.

**Fix:** In the store initializer, call `Appearance.getColorScheme()` and set `themeMode` accordingly if no persisted value exists.

---

### 17. `loadCalculation` Doesn't Trigger Result Display
**File:** `stores/lighting-store.ts`

`loadCalculation` restores all input fields and sets `lastCalculation` but does not set `showingPreview: true`. Loading a saved calculation restores internal state but the result card never appears — the user must hit Calculate again.

**Fix:**
```ts
set({ ..., lastCalculation: calc.result, showingPreview: true });
```

---

### 18. `beamWidth`/`beamHeight` Silently Default to 12m
**File:** `stores/lighting-store.ts`

In `calculate()`, empty beam fields default to `12.0` before unit conversion:
```ts
(parseFloat(state.beamWidth) || 12.0) * conv
```
The user sees no indication. In imperial mode this becomes `12.0 * FT_TO_M = 3.66m`, making imperial defaults produce different results than metric defaults. A phantom 12m beam silently shapes every result where the user left these fields empty.

**Fix:** Either require beam dimensions before allowing calculation, or display the applied default visibly in the UI.

---

### 19. `lastCalculation` Not Persisted — Result Lost on App Kill
**File:** `stores/lighting-store.ts`

`partialize` excludes both `lastCalculation` and `showingPreview`. If iOS kills the app while backgrounded, inputs rehydrate on relaunch but the result card is gone. The user must re-run the calculation to see results they already computed.

**Fix:** Add `lastCalculation` to `partialize`. The result object is already serializable JSON.

---

### 20. `savedCalculations` Grows Unbounded
**File:** `stores/lighting-store.ts`

Every saved calculation (containing a full `CalculationResponse` object with ~15 numeric fields) is prepended to an array that is entirely re-serialized to AsyncStorage on every mutation. With hundreds of entries, AsyncStorage rehydration on app launch — which blocks the JS thread — will become measurably slow.

**Fix:** Cap at 200 entries with FIFO eviction, or paginate using a lightweight index key in AsyncStorage.

---

### 21. Calculation ID Not Collision-Safe
**File:** `stores/lighting-store.ts`

```ts
id: `${timestamp}-${Math.floor(Math.random() * 100000)}`
```
Two saves within the same millisecond produce identical IDs. `deleteCalculation` filters by ID and would delete both. `addBlankFixture` in the simulation store uses 4 random alphanumeric chars — at best ~1.6M combinations with non-trivial collision probability under rapid fixture addition.

**Fix:** Use `crypto.randomUUID()` for all ID generation across both stores.

---

### 22. Demo Tour Re-Fires Mid-Session
**File:** `hooks/useFirstLaunch.ts`

For demo users, `setShouldShow(true)` fires synchronously in the effect without any AsyncStorage check. `markTourSeen` skips the AsyncStorage write (`if (userId && !isDemo)` guard). Every time `userId` or `isDemo` changes in the dependency array — including mid-session on logout → login — the effect re-fires and resets `shouldShow` to `true` even if the user just dismissed the tour.

**Fix:** Add a session-level `useRef` flag so the tour fires only once per component mount regardless of dependency changes.

---

### 23. `useFirstLaunch` Has No AsyncStorage Error Recovery
**File:** `hooks/useFirstLaunch.ts`

The hook calls `AsyncStorage.getItem(KEY).then(val => { if (val === null) setIsFirst(true); })` with no `.catch`. If AsyncStorage fails silently on a corrupted install, `isFirst` stays `false` — the onboarding modal never shows again, including on fresh installs.

**Fix:** Add a `.catch(() => setIsFirst(true))` fallback so failure-to-read defaults to showing onboarding.

---

### 24. LightSensorCard Renders on iOS as Permanently Broken UI
**File:** `hooks/useLightSensor.ts`

`checkAvailability()` is only called inside `if (Platform.OS === 'android')`. On iOS, `isAvailable` stays `false` permanently and `start()` silently no-ops. But `LightSensorCard` is rendered unconditionally on all platforms with no platform guard at the usage site. iOS users see a full sensor card with a "Start" button that does nothing.

**Fix:** At the usage site: `{Platform.OS === 'android' && <LightSensorCard />}`

---

### 25. QRScanner Hardcoded to Dark Theme
**File:** `components/QRScanner.tsx`

Uses `theme.colors.text` and `theme.colors.surface` directly from the static imported `theme` constant instead of `useThemeColors()`. Every other component in the codebase uses the reactive hook. The QR scanner always renders dark even when light mode is active.

**Fix:** `const colors = useThemeColors();` and replace all `theme.colors.*` references.

---

### 26. QRScanner Header Not Safe-Area Aware
**File:** `components/QRScanner.tsx`

`paddingTop: 60` is hardcoded on the scanner header. On iPhone 14 Pro+ with Dynamic Island, the status bar + island height exceeds 60pt, occluding the "Scan Fixture QR / Barcode" title.

**Fix:** `paddingTop: insets.top + 16` using `useSafeAreaInsets()`.

---

### 27. `setTimeout` setState After Scanner Unmounts
**File:** `components/QRScanner.tsx`

```ts
setTimeout(() => setHasScanned(false), 2000)
```
Normal flow closes the scanner within milliseconds of a successful scan. `setHasScanned` is called on an already-unmounted component. React 18 makes this a no-op but it holds a closure reference in memory and generates dev warnings.

**Fix:**
```ts
const timerRef = useRef<ReturnType<typeof setTimeout>>();
timerRef.current = setTimeout(() => setHasScanned(false), 2000);
// cleanup:
useEffect(() => () => clearTimeout(timerRef.current), []);
```

---

### 28. 4 Fluorescent Fixture Models Route to Wrong URLs
**File:** `utils/file-helpers.ts`

`L15T8/BLB`, `L9T8/BLB`, `L30T9/BLB`, `L15T9/BLB` are absent from `MANUAL_URLS` and `STORE_URLS`. The `startsWith('L')` fallback routes manual URLs to `sablelux-sableled-lamps/` and store URLs to `sableled-led-blb-lamps/` — third-party brand pages, not Wildfire product pages. "View Manual" and "Buy" buttons for these fixtures silently send users to the wrong site.

**Fix:** Add explicit entries for all four BLB models pointing to the correct Wildfire/SableLED pages.

---

### 29. `TypeScript ~5.9.2` Does Not Exist
**File:** `package.json`

TypeScript stable as of March 2026 is 5.7.x. The `~5.9.2` range will fail to resolve on a clean install in many package manager configurations, or silently pull a pre-release build.

**Fix:** Pin to `"typescript": "~5.7.2"`.

---

### 30. No `.env.example` in Repository
**File:** Project root

The app references `EXPO_PUBLIC_RORK_API_BASE_URL`, `EXPO_PUBLIC_TOOLKIT_URL`, `EXPO_PUBLIC_PROJECT_ID`, and `EXPO_PUBLIC_TEAM_ID` in the Expo start script. There is no `.env.example` documenting these. New contributors and deployment environments will silently fail all AI features with no error message explaining why.

**Fix:** Create `.env.example` listing all required variables with placeholder values and comments.

---

### 31. User ID Generation Uses Sequential Integers
**File:** `stores/auth-store.ts`

```ts
id: (users.length + 1).toString()
```
If any user is deleted and a new one is created, IDs collide. `deleteCalculation` filters by ID and would delete multiple records sharing the same ID.

**Fix:** Use `crypto.randomUUID()`.

---

### 32. `isAdmin` Is Entirely Client-Side
**File:** `stores/auth-store.ts`

`isAdmin` is set from a `role` field in AsyncStorage. Any user who can write to device storage can escalate to admin. There is no backend enforcement, making the admin gate cosmetic-only. For a local-only app this is acceptable — but admin-gated UI should be clearly marked as internal/demo-only, not exposed to end users as meaningful access control.

---

### 33. Tab Bar 6 Tabs on 320pt Screen
**File:** `app/(tabs)/_layout.tsx`

Six tabs ("Calculator", "Fixtures", "AI", "History", "Resources", "Profile") on an iPhone SE (320pt wide) gives each tab ~53pt. "Calculator" and "Resources" are 10+ character labels that clip. This is below Apple's 44pt minimum touch target recommendation.

**Fix:** Rename to shorter labels ("Calc", "Docs") or merge "History" into the calculator as a bottom sheet.

## 🟢 P3 — Polish / Technical Debt

### 34. Irradiance Formula Ignores Angular Falloff
**File:** `utils/lighting-calculator.ts`

```ts
irradianceMWm2 = peak_irradiance_mWm2 / throwDistance²
```
This is pure inverse square law for a point source in free space. Real UV fixtures follow a Lambertian or cosine distribution. Irradiance at an angled throw is lower than this model predicts — for close-range, high-power fixtures (VSP-120S at <3m), the model can overstate irradiance by 15–30%, understating actual safety risk.

**Fix:** Apply cosine correction: multiply irradiance by `cos(θ)` where `θ` is derived from vertical/horizontal geometry.

---

### 35. Safety Thresholds Have No Standard Referenced
**File:** `stores/lighting-store.ts` / `types/lighting.ts`

`getSafetyLevel` thresholds (DANGER >25,000 mW/m², WARNING >10,000, CAUTION >2,500) have no citation to ACGIH, IEC 62471, or any UV safety standard. For a professional UV lighting tool, thresholds should be tied to documented exposure limits.

**Fix:** Add inline comments citing the standard used. If these are Wildfire's own proprietary limits, state that explicitly.

---

### 36. `throwDistance` Has No Maximum Guard
**File:** `utils/lighting-calculator.ts`

The calculator rejects zero throw but has no upper bound. `verticalHeight = 10000` produces `0.00 mW/m²` with `100%` degradation — technically correct but meaningless. A practical ceiling (e.g., 100m) with a user-facing warning would improve UX and prevent nonsensical results.

---

### 37. 9 JS-Thread Animations on One Screen
**File:** `app/(tabs)/(home)/index.tsx`

`progressAnim`, `resultFadeAnim`, `resultSlideAnim`, `insightAnim`, `calcBtnScale`, `calcBtnGlow`, `stepContentAnim`, `stepSlideAnim`, `aiPulse` — nine `Animated.Value` refs, all driving JS-thread animations via the legacy `Animated` API. The glow animation loops indefinitely on the button. On mid-range Android, frame drops during step transitions are expected.

**Fix:** Migrate step transition and result fade to Reanimated 3 shared values + worklets (`useSharedValue`, `withTiming`, `useAnimatedStyle`).

---

### 38. `(any)` Cast on Calculation Result
**File:** `app/(tabs)/(home)/index.tsx`

```ts
const r = (lastCalculation as any).irradiance_report;
```
This completely defeats TypeScript's type narrowing. `CalculationResponse` is already a proper union type (`CalculationResult | CalculationError`).

**Fix:**
```ts
if ('irradiance_report' in lastCalculation) {
  const r = lastCalculation.irradiance_report;
}
```

---

### 39. Step Tab Labels 9px Font — Below Accessibility Minimum
**File:** `app/(tabs)/(home)/index.tsx`

`stepTabTitle` uses `fontSize: 9`. Apple HIG and Material Design both specify 11pt as the minimum legible label size. Users with any vision accommodation cannot read "Location", "Material", etc.

**Fix:** Raise to `fontSize: 11` and reduce `paddingHorizontal` by 2px to compensate.

---

### 40. Android Tab Bar `elevation: 16` Too High
**File:** `app/(tabs)/_layout.tsx`

`elevation: 16` is double Material Design's recommended 8 for bottom navigation. On older Android versions, this can cause rendering artifacts with content behind the tab bar.

**Fix:** Set to `elevation: 8`.

---

### 41. Mixed Package Managers (bun + npm)
**File:** `bun.lock` / `package.json`

A `bun.lock` (332KB) indicates Bun as the package manager, but the `package.json` scripts use npm syntax and Replit's agent runs npm. Mixed package managers in a team environment cause silent dependency version differences between environments.

**Fix:** Standardize on one package manager, delete the lockfile for the other, and document the choice in `README.md`.

---

### 42. Multi-Agent Commit Churn — Likely Dead Code Throughout
**Commit history**

Rork, Replit Agent, GitHub Copilot, and direct commits all made structural changes to the same files within hours of each other. Two "Fixed an error that prevented the app from starting" commits appear within 3 minutes of each other. The simulation screen, resources tab, and onboarding have each been rewritten 2–3 times — meaning orphaned imports and dead component variations are likely spread throughout.

**Fix:** Run `ts-prune` or `eslint-plugin-unused-imports` to identify all unreachable exports.

---

### 43. `RoomSimulation.tsx` 47KB Monolith
**File:** `components/RoomSimulation.tsx`

At 47,294 bytes — the largest file in the repo, larger than the entire calculator screen — with no sub-components, no extracted hooks, and no split between simulation view / controls / state. Metro parses and bundles this on every app start even when the simulate tab is not visited. The division-by-zero SVG bug (`59d2b05`) is buried inside it with no isolation boundary.

**Fix:** Extract into at minimum: `RoomCanvas`, `FixtureControlPanel`, `useRoomSimulation` hook, and `SimulationLayout`.

---

### 44. 4 Separate Persisted Zustand Stores on Startup
**Files:** `stores/*`

Auth, lighting, settings, favorites, and simulation each persist independently to AsyncStorage. That is 4–5 parallel async rehydration reads before the UI is fully usable. Each fires independently on startup.

**Fix:** Consolidate non-critical stores (favorites + simulation) into a single `'app-data'` persistence key, or lazy-load them on tab focus.

---

## ✅ What Is Done Well

- **Core math is clean** — `LightingCalculator` is a pure class with no side effects, straightforward to unit test
- **Zustand `partialize` used correctly** — ephemeral state (`isCalculating`, `isQRScannerOpen`) is correctly excluded from persistence
- **Theme system is consistent** — `useThemeColors()` used throughout; raw hex strings are absent
- **`useMemo` on styles** — `createStyles(colors)` is memoized in the calculator screen, preventing style object recreation on every render
- **Fixture data matches spreadsheet exactly** — 23 fixture models across VioStorm, Effects Master, UltraBlade, and fluorescent tube lines with no drift from the source TDS
- **QR scanner handles both JSON and raw string fixture codes** — graceful fallback logic in `handleQRScan`
- **Export pipeline architecture is correct** — platform-split (web blob vs native file) is the right approach; just needs the `await` fixes
