🔴 P0 — Ship-Blockers
1. Splash Screen Hides Before Auth Resolves — Login Flash on Every Cold Start
File: app/_layout.tsx

ts
useEffect(() => {
  initializeAuth();       // async — reads AsyncStorage
  SplashScreen.hideAsync(); // fires immediately, NOT awaited
}, [initializeAuth]);
initializeAuth() is async. The splash screen is dismissed before auth state is known. The router renders before authentication resolves, briefly showing the auth stack (/welcome or /login) even to logged-in users. This causes a visible white/dark flash or unauthorized screen flicker on every cold start.

Fix:

ts
useEffect(() => {
  initializeAuth().then(() => SplashScreen.hideAsync());
}, [initializeAuth]);
2. Auth Screens Permanently Dark — Light Mode Ignored
File: constants/theme.ts, app/(auth)/login.tsx, app/(auth)/register.tsx

ts
// constants/theme.ts
export const theme = {
  colors: darkColors,  // ← always darkColors, never lightColors
};
Both auth screens import theme directly and use theme.colors.text for UI elements:

ts
import { theme } from '@/constants/theme';
<ArrowLeft size={20} color={theme.colors.text} />
The useThemeColors() hook (used in tab screens) reads the live settingsStore.themeMode. The auth screens bypass this entirely. Every user who enables light mode sees dark-on-dark text in the back arrow, and the entire auth flow remains hardcoded dark. This is especially broken on iOS where system light mode is common.

Fix: Replace direct theme imports in auth screens with useThemeColors().

3. No Route Guard — Deep Links Can Expose Protected Screens
File: app/_layout.tsx

The root layout calls initializeAuth() but there is no route protection. Deep links or Expo Go direct navigation to a tab screen can resolve before initializeAuth() completes (AsyncStorage read ~50-100ms). A user arrives at /fixtures or /calculations unauthenticated for that brief window. If auth-store provides user data to screens that render it, a null deref crash is possible before the redirect fires.

Fix: Add a loading state check: render null or a loading view while initializeAuth() is pending before letting the router resolve.

4. warning and secondary Are Identical Colors in Dark Mode
File: constants/theme.ts

ts
darkColors = {
  secondary: '#F5A623',
  warning:   '#F5A623',  // ← exact same hex
}
The safety level system uses colors.warning for "caution" level calculations. Secondary UI elements (amber accents, section headers, fixture count chips) also use colors.secondary. In dark mode, a caution safety warning and a secondary decorative element are visually indistinguishable. A venue tech doing a safety review cannot tell if an amber badge means "caution UV level" or "secondary information." This is a safety-relevant color system flaw.

Fix: Assign a distinct amber-yellow to warning, e.g. '#EAB308' (which already exists as safetyYellow).

🟠 P1 — High Impact
5. Share Text Always Shows Meters — Imperial Users Get Wrong Units
File: app/(tabs)/calculations/index.tsx

ts
const shareText = `...Throw: ${(item.result as any).irradiance_report.throw_distance_m.toFixed(1)}m...`;
The share text hardcodes m and uses the raw throw_distance_m value regardless of unitSystem. The list view correctly converts with convertDistance(), but the shared text an imperial user sends always says e.g. "Throw: 6.1m" when their display shows "20.0ft". Every share from an imperial user is factually wrong.

Fix: Use convertDistance(throw_distance_m, unitSystem).toFixed(1) + distanceUnit(unitSystem) in the share string.

6. Project View Renders All Items Without Virtualization
File: app/(tabs)/calculations/index.tsx

tsx
// Project view mode:
{calcs.map((item, index) => (
  <View key={item.id}>{renderItem({ item, index })}</View>
))}
List view uses <FlatList> (virtualized). Project view switches to <ScrollView> with .map(), rendering every calculation card in every project group unconditionally. A user with 50+ saved calculations grouped by project renders all 50+ cards at once — no virtualization, no recycling. Each card has animations, safety dots, quick stats, and action chips.

Fix: Use FlatList for project view with ListHeaderComponent per group, or use SectionList to preserve virtualization.

7. Fixture Compare Always Uses Hardcoded 3m Throw
File: app/(tabs)/fixtures/index.tsx

ts
const throwDist = 3;
const result = calc.calculateRadiometricData(model, throwDist, 0, 6, 3);
The comparison modal subtitle says "At 3m throw distance, 6×3m target" — but this is fixed for every session. A user who has been designing for an 8m rig cannot compare fixtures at their actual throw. The comparison doesn't import useLightingStore() and has no way to pull the current calculator state.

Fix: Import useLightingStore and use verticalHeight (with a fallback of 3m if empty) so the comparison reflects the user's active configuration.

8. Enabling Biometric Requires No Proof of Biometric
File: app/(tabs)/profile/index.tsx

ts
{ text: 'Enable', onPress: () => setBiometricEnabled(true) }
When a user enables biometric login, the app just calls setBiometricEnabled(true) without performing an actual biometric authentication challenge. The standard security pattern is to authenticate once to prove the biometric is enrolled before enabling it. A user could enable biometric login even if Face ID or fingerprint isn't properly configured, then fail on every future login attempt.

Fix: Call authenticateWithBiometric() first. Only call setBiometricEnabled(true) if the challenge succeeds.

9. expo-file-system New API Usage May Silently Fail
File: app/(tabs)/calculations/index.tsx

ts
const { File, Paths } = await import('expo-file-system');
const file = new File(Paths.cache, `${item.name.replace(/\s+/g, '_')}.txt`);
file.create({ overwrite: true });
file.write(shareText);
This uses expo-file-system v2's experimental File class API. If the installed version is v1 (which doesn't export File or Paths), this destructuring succeeds with undefined values and silently falls to the catch block, showing the raw shareText in an Alert — a degraded but confusing experience. Additionally, file.write(shareText) is not awaited. If write() is asynchronous, shareAsync(file.uri) is called before the file is written.

Fix: Check the installed expo-file-system version. Either add await to async calls, or use the v1 FileSystem.writeAsStringAsync() API which is stable and universally supported.

10. Category-Level Control Badge Uses fixtures — Wrong for EM Series
File: app/(tabs)/fixtures/index.tsx

ts
<Text>{getFixtureControlType(fixtures)}</Text>
The control type badge displayed at the category header level comes from the first fixture in the group. For the EM category, fixtures is determined by insertion order in LightingCalculator.FIXTURES. If EM-44V appears first, getFixtureControlType returns 'On/Off (Mains)' (the pre-existing bug). The entire EM category is then labeled "On/Off (Mains)" in the UI, even though some EM models are DMX. This label appears prominently in the colored badge at the top of every EM section.

11. Fixture Stats Card Shows Hardcoded 23 for Fixture Count
File: app/(tabs)/profile/index.tsx

tsx
<Text style={styles.statValue}>23</Text>
<Text style={styles.statLabel}>Fixtures</Text>
This is a hardcoded literal. The actual count should be LightingCalculator.getFixtureModels().length. File: app/(auth)/login.tsx

ts
console.log('[Login] Apple:', appleOk, 'Biometric:', bioOk);
This logs auth capability detection to the console in production builds. Not a direct security risk (it logs boolean availability, not credentials), but it's a debug artifact that should be removed or gated with if (__DEV__).

13. Demo Account Credentials Hardcoded as String Literals
File: app/(auth)/login.tsx

ts
const success = await login('demo@example.com', 'password123');
Credentials are inline string literals in the component. If the demo account changes, this requires a code change with no compile-time signal. Should be extracted to a constant in a config file, or ideally handled by the auth store as a dedicated loginDemo() method.

14. Register Screen: No Email Format Validation
File: app/(auth)/register.tsx

ts
if (!name || !email || !password || !confirmPassword) { ... }
email only checks for non-empty. 'notanemail', 'a@b', and '@@@' all pass. The auth-store's register() function stores the email as-is. Malformed emails break the login flow later when the user tries to sign in (they typed the right string but it was wrong to begin with).

Fix: Add /^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email) validation before attempting registration.

15. Confirm Password Field Missing showPasswordToggle
File: app/(auth)/register.tsx

tsx
<Input label="Password" ... showPasswordToggle />
<Input label="Confirm Password" ... />  {/* no showPasswordToggle */}
Users can reveal the first password field to verify what they typed, but not the confirm field. They must type the confirm password blindly. This increases registration abandonment on mobile keyboards.

16. Password Minimum Only 6 Characters — No Complexity Check
File: app/(auth)/register.tsx

ts
if (password.length < 6) { Alert.alert('Error', 'Password must be at least 6 characters'); }
'aaaaaa' and '123456' pass validation. For a professional B2B app used in UV safety contexts containing saved calculation data, a minimum of 8 characters with at least one number or symbol is industry standard (NIST SP 800-63B).

17. handleOpenUrl and renderIcon Use Dynamic Imports / any Casts

Profile screen dynamically imports expo-web-browser inside a press handler (same issue as expo-clipboard noted in Level 3). 18. Resource External Links Silently Swallow Errors
File: app/(tabs)/resources/index.tsx

ts
Linking.openURL(item.url).catch(() => {});
The empty catch block means if a resource URL is dead, malformed, or blocked (e.g., user has no browser), nothing happens. No toast, no error state, no retry. The user taps a resource link and nothing happens.

Fix: Add a user-facing fallback: catch(() => Alert.alert('Could not open link', item.url)).

19. loadCalculation + Alert Tells User to Manually Navigate

ts

After loading a calculation, the user is interrupted by an Alert telling them to manually switch tabs. The loaded calculation is now in the store, but the user must dismiss the alert and then tap the Calculator tab manually.

21. pendingDemoLogin Auto-Login on Onboarding Dismiss Is Surprising
File: app/(auth)/login.tsx

If a first-time user taps "Try Demo Account" while the onboarding modal is showing, pendingDemoLogin is set to true. When they dismiss the onboarding modal (by reading it and closing it), executeDemoLogin() fires automatically. A user who reads the whole onboarding and then closes it is unexpectedly logged in without any second confirmation. This is by design intent but may surprise users.
